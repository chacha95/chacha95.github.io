# MLflow란?

PyTorch는 연구에서 생산에 이르는 워크 플로를 가속화 한 PyTorch 1.0이 도입 된 이후로 빠르게 발전해 왔습니다. paperswithcode.com/trends에 표시된 것처럼 연구의 추진력을 살펴보면 연구 커뮤니티가 PyTorch를 선택한 도구로 채택했음을 알 수 있습니다. 반대로 Lyft Level 5, Microsoft, AstraZeneca 및 기타 여러 기업과 같은 전 세계의 기업 및 프로덕션 사용자는 PyTorch를 대규모로 배포하여 실제 제품 가치를 실현하고 있습니다. MLOps와 머신 러닝의 엔드 투 엔드 라이프 사이클 관리는 이러한 커뮤니티를 연결하는 중요한 다리이며, PyTorch와 MLOps에서 개발 된 최첨단 AI의 결합은 최첨단 연구의 영향을 제품에 가져올 강력한 힘입니다.

오늘 우리는 다음에 대한 지원을 포함하여 PyTorch를 사용한 MLflow 사용에 대한 종단 간 지원을 가능하게하는 여러 기술적 기여를 발표합니다. PyTorch Lightning을 통한 자동 로깅; 새로운 배포 플러그인을 통한 TorchServe 통합; HuggingFace Transformers를 대상으로하는 샘플 엔드-투-엔드 워크 플로우. Databricks / MLflow 핵심 관리자의 기술 팀 구성원과 Facebook 내의 PyTorch 핵심 개발 팀 간의 공동 작업으로 이루어진이 작업은 시작에 불과합니다. 앞으로 몇 달 동안 더 많은 기술적 기여를 기대합니다.
MLOps의 부상

Wikipedia에 따라 MLOps는 다음과 같이 정의됩니다.

"머신 러닝"과 "운영"의 조합은 프로덕션 ML (또는 딥 러닝) 수명주기를 관리하는 데 도움이되는 데이터 과학자와 운영 전문가 간의 협업 및 커뮤니케이션을위한 관행을 말합니다.

MLOps에는 모델 개발, A / B 테스트, 지속적인 통합 / 제공, 모니터링 등을 포함하여 라이프 사이클의 다양한 단계에서 사용자가 모델을 관리하는 방법이 포함됩니다. 지난 10 년 동안 AI 연구 붐의 초기 단계에서는 일반적으로 그렇지 않았습니다. 어느 정도 규모로 운영하지 않는 한 커뮤니티의 관심 영역입니다. ML의 확산과 관련 제품에 미치는 영향이 느껴지면서 MLOps의 주제는 전면 중심이며 커뮤니티는 활발하고 참여도가 높습니다. 그러나 생산 패러다임에 대한 연구를보다 광범위하게 사용할 수 있도록 처리해야 할 과제가 있으며 여전히 수행해야 할 작업이 있습니다.

우리가 관찰 한 몇 가지 과제 :

    도구 체인 다양성 : ML의 워크 플로는 데이터 준비 및 분석, 모델 학습, 최적화, 배포 등을 포괄합니다. 도구를위한 공간은 ML 자체와 마찬가지로 세분화되고 계속 진화합니다.
    실험 추적은 어렵습니다. ML 모델 개발에는 실험 과정이 내재되어 있습니다. 확장으로 특정 제품 요구 사항 집합을 충족하는 모델을 개발하려면 훈련 된 매개 변수, 코드 버전, 데이터 및 모델의 조합을 폭발적으로 확장해야 할 수 있습니다.
    재현성은 어렵습니다. ML 시스템 내에서 변수의 수와 무작위성의 원인을 고려할 때 결과를 재현하기가 어렵습니다. 일반적으로 동일한 코드를 다시 작동 시키려면 문제를 디버깅하거나 재현 가능한 워크 플로를 공유하는 것이 여전히 어려운 과제입니다.
    배포는 여전히 쉽지 않습니다. 프로덕션 환경은 배치 및 스트리밍 추론, 기기 추론을 포함하여 단편화되고 다양하며 일반적으로 위험이 낮은 방식으로 배포 할 수있는 표준 방법이 없습니다.
    지속적이고 반복적 인 프로세스 : 기존 소프트웨어 개발과 달리 ML 모델을 구축하고 관리하는 것은 지속적이고 반복적 인 프로세스입니다. 정확성과 같은 측정 항목을 최적화하려면 지속적인 실험이 필요합니다. 모델은 데이터, 사용자 행동의 변화로 인해 시간이 지남에 따라 드리프트 될 수 있으며 시간이 지남에 따라 좋은 결과를 얻으려면 전반적인 교육 / 재교육 / 피드백 루프가 필요합니다.
    클라우드 독립성 : 관리 형 서비스는 훌륭하지만 몇 가지 단점도 있습니다. 여기에는 사용자 정의, 유연성 및 공급 업체 종속의 잠재적 부족이 포함됩니다. 제품이 규모에 도달하기 시작하면 이러한 요소의 중요성이 점점 커집니다.

MLflow + PyTorch = 🧡

PyTorch의 프로덕션 플랫폼에 대한 종단 간 탐색을 가능하게하는 첫 번째 단계로, 우리는 MLflow 커뮤니티와 협력하여 프로젝트를 더욱 긴밀하게 통합했습니다. 여기에는 자동 로깅 (명시적인 로그 문이 필요없는 로그 메트릭, 매개 변수 및 모델)에 대한 지원 추가 및 성능 모델 추론을 지원하기 위해 배포 플러그인을 제공하는 TorchServe 모델 통합이 포함되었습니다. 모든 것을 하나로 묶기 위해 사용자가 시작하는 데 활용할 수있는 Hugging Face Transformers를 기반으로 한 엔드 투 엔드 NLP 워크 플로도 제공합니다.

또한 Python이없는 프로세스에서 배포를 위해 모델을 직렬화 및 최적화하는 방법 인 TorchScript 및 더 크고 빠른 모델 개발을 지원하기위한 분산 교육도 사용할 수 있습니다. 다음 섹션에서는 이러한 각 영역에 대한 세부 정보 중 일부를 살펴볼 것입니다.



자동 로깅

Autolog를 사용하면 ML 모델 빌더가 MLflow의 PyTorch 모델에서 매개 변수와 메트릭을 자동으로 기록하고 추적 할 수 있습니다. 핵심 모델 메트릭 로깅 및 MLflow 실험 추적에 대한 모범 사례를 기반으로 자동 로깅에 대한 지원을 추가하기 위해 PyTorch Lightning을 학습 루프로 사용했습니다. Autolog 기능은 최적화 프로그램 이름, 학습률과 같은 매개 변수를 자동으로 기록합니다. 훈련 손실, 검증 손실, 정확도와 같은 메트릭; 아티팩트 및 체크 포인트 형태의 모델. 조기 중지를 사용하면 모델 체크 포인트, 조기 중지 매개 변수 및 메트릭도 기록됩니다. 또한 사용자는 F1 점수와 같은 사용자 지정 지표를 유연하게 기록 할 수 있습니다.
모델 아티팩트 및 TorchScript 지원

이제 NLP 모델의 어휘 파일과 같은 추가 모델 아티팩트를 저장할 수있는 기능을 사용하여 Eager 및 TorchScript 모드에서 PyTorch 모델을 저장하고로드 할 수 있습니다.

TorchScript는 ML 애플리케이션 및 프로덕션 모델 제공에 권장되는 Python 언어의 정적 하위 집합입니다. 프로덕션을 위해 모델을 준비하는 최적화 단계의 일부로 모델을 TorchScript 형식으로 변환 한 다음 프로덕션에 배포합니다. 이제 MLflow를 사용하여 TorchScripted 모델을 저장하고로드 할 수 있습니다.

모든 아티팩트는 MLflow MLmodel 번들의 일부로 저장되므로 MLflow 모델 레지스트리의 개발, 스테이징에서 프로덕션 단계로 이동할 때 모델의 여러 단계를 쉽게 관리 할 수 있습니다.
TorchServe를 통한 배포

MLflow TorchServe 플러그인을 통해 사용자는 이제 모델 제공까지 전체 MLOps 수명주기를 얻을 수 있습니다.

TorchServe는 다중 모델 제공, 모델 버전 관리, A / B 테스트, 모델 메트릭을 지원하여 대규모 PyTorch 모델 배포를 가속화하는 PyTorch 모델 제공 라이브러리입니다. 객체 감지 및 텍스트 분류와 같은 일반적인 사용 사례를위한 기본 핸들러와 빠르게 시작할 수있는 모델 동물원이 함께 제공됩니다.

배포를 생성 및 관리하고 이에 대한 추론을 실행하기 위해 명령 줄 인터페이스 또는 Python API를 사용할 수있는 옵션이 있습니다. TorchServe의 로컬 및 원격 설치 모두 플러그인에 의해 지원되므로 개발 라이프 사이클의 여러 단계에 대해 최적의 워크 플로우를 가질 수 있습니다.
재현 가능한 실행을위한 ML 프로젝트

모든 샘플은 익숙한 ML 프로젝트 패키지와 함께 제공되므로 누구나 쉽게 재현 가능한 실행을 시작할 수 있습니다.
분산 교육

분산 학습이 필요한 대형 모델에 대해 동일한 익숙한 흐름을 사용할 수 있습니다. 팀이 쉽게 시작할 수 있도록 PyTorch 분산 데이터 병렬 교육을 사용하기위한 전체 워크 플로가 제공되었습니다.
PyTorch + MLflow + TorchServe에 대한 MLOps 워크 플로
